Engineering Intent

AI delivery is controlled by explicit workflows, typed boundaries, and durable memory

In this repo, workflow is not a meeting ritual. It is a defined sequence of skills, quality gates, and memory updates that must close before a change is considered done.

Control Inputs

docs/ standards + canonical skills in .agents/skills.

Control Execution

Effect-based implementation paths with typed contracts and explicit runtime layers.

Control Closure

Checks pass, risks reviewed, docs synced, and workflow memory event appended.

Workflow Definition

What “workflow” means in this repository context

Workflow = repeatable, enforceable delivery loop

A workflow in Content Studio is a scoped operating protocol: choose the right skills, follow repo patterns, run mandatory checks, and persist evidence in workflow memory.

Activation

  • Triggered by request type, risk profile, and touched surfaces.
  • Examples: Feature Delivery, TanStack + Vite, PR Risk Review.

Execution

  • Implement in vertical slices.
  • Enforce architecture/auth/safety primitives and typed error handling.

Completion

  • Validation gates pass.
  • Memory event IDs captured in delivery notes.

Spec-Driven Design

How requirements govern implementation and prevent spec drift

Spec workflow

Propose behavior in docs/master-spec.md -> Run pnpm spec:generate -> Reference SPEC-* IDs in PR/tests -> Gate with pnpm spec:check

Generated artifacts include API/domain/data/UI surfaces under docs/spec/generated/.

Why this matters

  • Behavior changes require explicit design review before merge.
  • Generated snapshots expose contract and route drift quickly.
  • PRs become traceable from requirement ID to code and tests.

Spec-Driven Example (Today)

Illustrative real-flow example using current repo structure

Example requirement

Add requirement SPEC-DOCUMENT-### (ID pattern from repo rules): “Document retry is allowed only for failed documents; otherwise return protocol-mapped error.”

Execution steps

1. Update non-generated spec text in docs/master-spec.md
2. Run pnpm spec:generate and inspect generated artifacts
3. Implement behavior in domain + contract + handler layers
4. Add/adjust tests and invariant coverage if rule is non-negotiable
5. Run pnpm spec:check + repo gates before merge

Likely files touched

  • Spec source: docs/master-spec.md
  • Generated spec: docs/spec/generated/*
  • Use case: packages/media/src/document/use-cases/retry-processing.ts
  • Contract: packages/api/src/contracts/document.ts
  • Handler: packages/api/src/server/router/document.ts
  • Tests: use-case + integration + invariants where applicable

Type System

TypeScript + Effect types are part of the guardrail system

Type Boundary Rule Why It Exists
Effect control flow Effect<A, E, R> keeps success/errors/dependencies explicit. Prevents hidden runtime assumptions in agent-authored logic.
Error unions Let Effect infer errors; export derived aliases from use cases. Keeps handler and tests aligned with real failure surface.
Layer environments Derive env from layers: Layer.Layer.Context<typeof Live>. Avoids drift between runtime wiring and type declarations.
Streaming contracts Use concrete stream chunk types, not unknown. Eliminates downstream casting and runtime ambiguity.

Effect Decision

Why this repo standardized on Effect TS over NestJS

Concern Effect TS Advantage Operational Outcome
Control flow typing Explicit typed success, typed errors, typed requirements. More deterministic review and fewer hidden runtime failures.
Dependency wiring Layers/service tags are explicit values. Easier diffing and targeted runtime test setup.
Error contracts Tagged errors map via centralized protocol handler. Stable API error behavior across handlers.
Testability Use cases run with lightweight layers/mocks. Fast iteration with strong safety checks.

Source: docs/architecture/effect-vs-nestjs.md.

Checks & Gates

Linting, invariants, tests, and spec checks in one matrix

Gate Command What It Protects
Type safety pnpm typecheck Cross-package TS correctness and contract consistency.
Linting pnpm lint Pattern enforcement (no unsafe casts/throws/payload ambiguity).
Test suite pnpm test Unit/integration/frontend regressions.
Invariant tests pnpm test:invariants Safety primitives, error mapping, architecture constraints.
Spec drift pnpm spec:check Behavior changes without updated spec artifacts.
Build integrity pnpm build Compilation/runtime packaging regressions.
Skill quality pnpm skills:check:strict Skill metadata/path/symlink drift.
Memory coverage pnpm workflow-memory:coverage:strict Missing workflow classes in monthly memory logs.
  • If spec:check fails: either regenerate artifacts or update the intended behavior/spec text.
  • If test:invariants fails: you changed a non-negotiable pattern and must re-align implementation or invariant rules intentionally.
  • If lint/typecheck fail: the workflow blocks merge until structural and type-level issues are resolved.
