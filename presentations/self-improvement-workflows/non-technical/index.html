Definition

What a workflow means in this repo

Workflow = repeatable route from request to verified outcome

In Content Studio, a workflow is a prescribed sequence: intake, implement, verify, review risk, and persist learning. It is designed for consistency and auditability.

Spec-Driven Delivery

Behavior is defined in spec before final merge

How it works

  • docs/master-spec.md is the canonical behavior source.
  • Generated spec surfaces are refreshed from code.
  • pnpm spec:check fails when behavior and spec drift apart.

Business value

  • Fewer surprises during release and QA.
  • Clear requirement traceability in PR discussions.
  • Stronger alignment between product intent and shipped behavior.

Brief Example

How spec-driven development works today in practice

Request: change document retry behavior -> Update requirement text in docs/master-spec.md -> Run pnpm spec:generate
Implement code + tests -> Run checks (typecheck, lint, test, test:invariants) -> Pass pnpm spec:check before merge

The key idea: requirements and code are verified together. If behavior changes but spec artifacts do not, the merge should fail.

Architecture Choice

Why type-heavy Effect TS patterns are used

Plain-language summary

The repo favors Effect TS because it makes dependencies and errors explicit in types. That gives clearer feedback and safer AI-authored changes.

Practical impact

  • More predictable behavior under change.
  • Smaller, faster tests with explicit runtime setup.
  • Less hidden framework magic during debugging.

Checks and Protections

What quality gates run to prevent regressions

Check Example Command Purpose
Type system checks pnpm typecheck Detects contract and compile-time issues across packages.
Linting checks pnpm lint Blocks unsafe or off-pattern code constructs.
General tests pnpm test Catches behavior regressions.
Invariant tests pnpm test:invariants Enforces non-negotiable safety and architecture rules.
Spec checks pnpm spec:check Ensures documented behavior matches implementation.
