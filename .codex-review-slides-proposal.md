# Slides Feature Proposal — Content Studio

## Issue #123: PowerPoint/Slide Deck Generation

### Vision
Allow users to turn their documents (research, uploads, URLs) into polished, interactive slide presentations. Users can chat with AI to:
- Generate an initial slide deck from document content
- Make global changes (themes, styles, layouts)
- Edit individual slides (content, images, structure)
- Reference existing documents and infographic images
- Export to HTML or PowerPoint

### Inspiration
[zarazhangrui/frontend-slides](https://github.com/zarazhangrui/frontend-slides) — generates self-contained HTML slide decks using Claude with 10 curated style presets. Zero-dependency single HTML files with navigation, animations, and responsive design.

---

## Current Architecture Context

Content Studio has these existing entities: Document, Podcast, Voiceover, Infographic, Persona. Each follows a rigid pattern:
- DB schema with branded IDs, Effect serializers
- Media package with repos/use-cases/errors
- oRPC contracts + handlers
- Job queue integration for async generation
- SSE events for real-time updates
- Frontend: TanStack Router routes + feature folder with hooks/components

The **Infographic** entity is the closest analog — it has style properties, async image generation, version history, and an approval workflow.

The **Chat** feature already has streaming AI endpoints (research, persona, writing assistant) using Vercel AI SDK types.

## Key Design Decisions to Debate

### 1. Slide Representation Format

**Option A: Self-contained HTML (like inspiration repo)**
- Pros: Simple storage (single blob), instant preview, export-ready, AI can generate/edit directly
- Cons: Hard to do per-slide editing, no structured data for slide-level operations, harder to extract/merge content

**Option B: Structured JSON with per-slide data**
```typescript
interface SlideDeck {
  id: string;
  title: string;
  theme: ThemeConfig;
  slides: Slide[];
}
interface Slide {
  id: string;
  order: number;
  content: SlideContent; // title, body, bullets, images, layout
  notes?: string;
  sourceDocumentIds?: string[];
}
```
- Pros: Per-slide editing, structured operations, easy reordering, document references per slide
- Cons: Need a renderer (React component or HTML generator), more complex data model, two representations (data + rendered)

**Option C: Hybrid — structured JSON stored, HTML rendered on demand**
- Store slides as structured JSON for editing
- Render to HTML for preview/export via a template system
- AI operates on the JSON structure but can also regenerate HTML

### 2. AI Integration Architecture

**Option A: New dedicated chat endpoint (like writingAssistant)**
- Add `slidesAssistant` to chat contract
- Streaming chat that understands slide structure
- Can make global (theme) and local (slide content) changes
- Returns structured tool calls that modify the slide JSON

**Option B: Tool-use pattern — AI returns slide mutations**
- AI chat returns structured commands: `{action: "updateSlide", slideIndex: 2, content: {...}}`
- Frontend applies mutations to local state
- More predictable, testable, but less conversational

**Option C: Direct HTML generation/editing**
- AI generates/modifies raw HTML per slide or per deck
- Simpler AI integration (just text in, text out)
- But: loses structure, harder to do precise edits

### 3. Document Relationship Model

**Option A: Many-to-many with junction table**
- `slide_deck_document` junction table
- Full referential integrity
- Can track which slides use which documents
- Adds DB complexity

**Option B: JSONB array of document IDs on slide deck**
- `sourceDocumentIds: string[]` on the deck
- Simpler schema, no joins needed
- Loses per-slide document tracking
- No referential integrity

**Option C: Per-slide document references in the JSON structure**
- Each slide in the JSON can reference document IDs
- No separate DB table needed
- Document references live in the slide content structure
- Trade-off: no DB-level querying of relationships

### 4. Scope & Complexity Tiers

**Tier 1 — MVP (Minimal)**
- New `slide_deck` entity following existing patterns
- AI generates a complete HTML slide deck from selected documents
- Store as single HTML blob (like infographic stores an image)
- Basic chat to regenerate/edit the whole deck
- Export as HTML file download
- No per-slide editing, no PowerPoint export
- ~Same complexity as infographic feature

**Tier 2 — Structured (Medium)**
- Structured JSON slide model with per-slide data
- React-based slide renderer/previewer
- Per-slide AI editing via chat
- Theme system with presets
- Document references per slide
- Version history (like infographic)
- PowerPoint export via a library (pptxgenjs or similar)

**Tier 3 — Full Interactive (High)**
- WYSIWYG slide editor
- Drag-and-drop reordering
- Real-time collaborative editing
- Custom theme builder
- Image search/generation per slide
- Presenter mode with notes
- Animation system

### 5. Frontend Rendering Approach

**Option A: iframe sandbox**
- Render HTML slides in an iframe
- Full isolation, supports any HTML/CSS
- But: communication between iframe and React app is clunky

**Option B: React slide components**
- Build slide renderer as React components
- Full integration with app state
- But: need to maintain a rendering system, harder to export

**Option C: Hybrid — React for editing, HTML for preview/export**
- Edit structured data in React forms/panels
- Generate HTML for preview pane (rendered in iframe or shadow DOM)
- Export the generated HTML directly

### 6. Theme/Style System

**Option A: Predefined presets only (like inspiration repo)**
- 8-10 curated themes
- User picks one, AI applies it
- Simple, opinionated, high quality
- Limited customization

**Option B: Style properties (like infographic)**
- Key-value style properties with AI interpretation
- Flexible but unpredictable results
- Users can tweak individual properties

**Option C: CSS variables theme system**
- Themes defined as CSS custom property sets
- Presets + user customization
- Predictable rendering, easy to override
- More engineering effort upfront

## Engineering Concerns

1. **14-layer registration**: Adding a full entity touches DB, media, queue, contracts, router, worker, SSE, frontend routes, features, dashboard, sidebar, SSE handlers, activity log, testing. This is a LOT of boilerplate.

2. **AI token costs**: Generating/editing full HTML slide decks involves large prompts. Structured JSON + templates could reduce token usage.

3. **PowerPoint export**: Libraries like `pptxgenjs` add bundle size. Could be a server-side operation instead.

4. **Image references**: Slides should be able to reference infographic images already generated. Need storage key access pattern.

5. **Chat context window**: Per-slide editing requires the AI to have context of the full deck + current slide + source documents. Could get large.

6. **Rendering fidelity**: HTML slides in a browser look different from PowerPoint slides. Managing expectations is important.

---

## Product Concerns

1. **User workflow**: How does a user start? From a document? From scratch? From a template?
2. **Editing UX**: Is chat-based editing sufficient, or do users need direct manipulation?
3. **Quality bar**: AI-generated slides need to look professional. Generic outputs will disappoint.
4. **Differentiation**: What makes this better than just using Google Slides + AI?
5. **Export formats**: HTML is easy but niche. PowerPoint is expected. PDF is useful.
6. **Collaboration**: Is this single-user or does it need sharing/approval workflows?

---

## Recommended Approach (for debate)

Start with **Tier 1.5** — between MVP and Structured:

1. **New `slide_deck` entity** following existing patterns (branded ID `sld_`)
2. **Structured JSON storage** for slides (not raw HTML) — enables per-slide operations
3. **Predefined theme presets** (8-10, curated) — no custom theme editor
4. **New `slidesAssistant` chat endpoint** — streaming, can modify individual slides or global theme
5. **React slide renderer** for preview (converts JSON → visual slides)
6. **HTML export** first, PowerPoint export as follow-up
7. **Document references** as JSONB array on the deck (not per-slide junction table)
8. **Version history** on regeneration (like infographic)
9. **No WYSIWYG** — AI chat is the primary editing interface

This balances simplicity with the interactive editing vision while following existing patterns.
